    Engenharia de Requisitos (Existe uma matéria só pra isso)
        Requisitos
            São as descrições do que o sistema deve fazer, os serviços que oferece e as restrições a seu funcionamento.

        Engenharia de Requisitos
            Processo de descobrir, analisar, documentar e verificar esses serviços e restrições.
                - Observação
                - Pesquisa, Formulários
                - É importante saber quem faz
                - Evite ser ortodoxo (afirmar que já fez x e que o y será igual)
                - O que não vai ter?
                - Eu posso acompanhar a dinâmica do processo?
                - Sempre vá em par, evite sempre que possível ir sozinho

        Mitos sobre Requisitos
            - Eu começo a programar enquanto você descobre o que o cliente quer
            - Vai ser fácil, O Cliente ou o usuário sempre sabe o que quer
            - Tranquilo, Requisitos são fáceis de obter

        Engenharia de Requisitos (Níveis)
            Nível de Detalhamento 
                = 
                    + Requisito de usuários (requisitos abstratos de alto nível) (Falar por cima)
                        - Devem ser escritos de uma maneira compreeensível para o usuário, sem termos técnicos, notações, etc
                        - Utiliza-se linguagem natural com tabelas, formas e diagramas intuitivos
                        Exemplo:
                        - O sistema deve gerar a folha de pagamento do funcionários no último dia útil do Mês
                    + Requisitos de sistemas (descrição detalhada do sistema) 
                        - São versões expandidas dos requisitos do usuário, detalhando e explicando o comportamento externo do sistema e suas restrições operacionais
                        - Utiliza-se linguagem natural ou outras anotações
                        Exemplo:
                        - No último dia útil de cada mês deve ser gerado um resumo das faltas e atrasos de cada colaborador, verificando a quantidade de dependentes e bonificação
                        - Calcular o valor do salário bruto considerando o cargo e nível de cada colaborador, calcular descontos e proventos
                        - Imprimir o holerite e gerar um arquivo para lançamentos para creditar na conta corrente do colaborador e debitar da conta corrente da empresa.

                    O Nicho principal deve ser feito o Nível detalhado, o cérebro da aplicação.
        
        Especificação de Requisitos
            Os requisitos devem ser claros, sem ambiguidades, de fácil compreensão, completos e consistentes

        Desafios
            - Imprecisão na especificação é a causa de muitos problemas
            - Desenvolver pode interpretar um requisito de uma maneira diferente da necessidade do cliente
            - Características principais
                - Completude: Devem incluir a descrição de todos os itens requeridos
                - Consistência: Não deve haver conflitos ou contradições

    
        Deixar de atender um requisito não funcional pode inutilizar o sistema

        Classificação:
            Requisito de produto: Comportamento do software
            Requisito organizacional: Derivado das políticas e procedimentos da organização do cliente e do desenvolvedor
            Requisitos externos: Todos que derivam de fatores externos. Exemplo: Reguladores legais, éticos


        Requisitos Funcionais (RF)
            Tudo o que vou codar é funcional
            Declarações de serviços que o sistema deve fornecer, como o sistema deve reagir a entradas específicas e como deve se comportar em determinadas situações.
            
        Requisitos Não Funcionais (RNF)
            - Exemplo: De que horas a que horas seu sistema vai ficar no ar? Quando vai fazer o backup? Qual a linguagem? Qual navegador? Que horas vai atualização do sistema? Em que momento?
            Restrições ou funções oferecidas pelo sistema, como restrições de timing, processo de desenvolvimento, e restrições impostas pelas normas
